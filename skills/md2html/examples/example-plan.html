PLAN

Migrating the notification system from polling to webhooks

Cut latency from ~3 minutes to under 5 seconds and reduce DB load by 60% by replacing a 5-minute cron with Kafka + a webhook router.

example-plan.md 2026-05-14 ~3 min read

Background

The notification-worker runs a cron job every 5 minutes that scans the events table for rows with status = 'pending' and dispatches notifications. This approach has three problems:

  • End-to-end latency averages 2.5 minutes, with p99 reaching 6 minutes.
  • Each scan reads ~50k rows, consuming 30% of CPU on the DB primary.
  • During traffic spikes the queue backs up — some events take 30+ minutes to process.

Quarterly KPI

The team is committed to dropping notification latency below 10s this quarter. This plan is how we deliver on that.

Goals

After rollout:

  • p99 notification latency: 360s → <5s
  • DB query load reduced by 60% (measured via pg_stat_statements)
  • Zero downtime, zero event loss during migration

Proposed architecture

Producers (order-service, payment-service, …) emit events to Kafka. notification-router subscribes to topic events.* and routes each event to the appropriate channel (email/push/SMS) via a webhook to notification-worker. The worker delivers the notification and writes an audit log to Postgres.

Security

The webhook between router and worker runs inside the same VPC, not over the public internet. Signature verification is not required — mTLS is sufficient.

Message bus choice

Three main candidates:

Kafka

High throughput, cluster already deployed. 7-day retention enables replay when the worker fails. Downside: the team isn't fully familiar with consumer-group rebalancing.

RabbitMQ

Simple and easy to operate. But no replay, and messages can be lost if a consumer dies before acking.

AWS SQS

Managed, no ops burden. Costs about $0.4 per million messages — roughly $20/month at 50M. Slight vendor lock-in.

Decision

Go with Kafka. The infrastructure already exists, and replay capability is essential for avoiding event loss during worker restarts.

Risks

Lost webhook on worker restart

We must implement an idempotency key plus exponential backoff retry in the router so events are never silently dropped.

Double-write inconsistency (week 2)

A Kafka emit can fail after the DB insert succeeds. Mitigation: use the transactional outbox pattern.

Do not drop the events table before week 5

If we hit a bug in week 4 and need to roll back, that table is our only restore source.

Pros & cons of the webhook approach

✓ Pros

  • Real-time delivery, no polling overhead
  • DB load drops drastically
  • Scales horizontally with Kafka partitions

✕ Cons

  • Retry and idempotency need careful handling
  • Harder to debug than cron (state lives in the queue)
  • Two code paths to maintain for four weeks

Retry policy

Exponential backoff and dead-letter handling

When a webhook call fails, the router retries with exponential backoff: 1s, 4s, 16s, 64s, 256s, 1024s. After six failures the event is pushed to dead-letter topic events.dlq and oncall is paged.

Idempotency key:

idempotency_key = sha256(event_id + channel + recipient)
// store in Redis with 24h TTL
// worker checks before delivering

Summary

Key point

Switching from polling to webhooks cuts 99% of latency and 60% of DB load in exchange for a moderate increase in operational complexity. A five-week plan with one week of monitoring before cutover keeps rollback safe.
